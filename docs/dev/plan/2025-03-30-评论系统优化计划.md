# 评论系统优化计划

> 本文档记录动态页面评论展示优化的实现计划，参考QQ空间风格设计。

## 概述

**创建时间**: 2025-03-30
**优先级**: 高
**影响范围**: frontend/client

---

## 需求描述

优化动态页面的评论展示系统，参考QQ空间的动态和评论交互风格：

### 已确认的需求细节

| 需求项 | 确认结果 |
|--------|----------|
| 通知入口位置 | 仅在动态页面右上角显示通知图标 |
| 后端API | 需要新增通知相关接口和数据库表 |
| 评论UI风格 | 扁平化卡片式（回复内嵌在主评论下方） |
| 评论排序 | 正序（最早发表的评论在前） |
| 通知类型范围 | 仅包含评论和回复相关的通知 |

### 功能需求

1. **评论功能**
   - 支持对动态发表评论（一级评论）
   - 支持对评论进行回复（嵌套回复，无限层级）
   - 支持删除自己的评论（级联删除所有子评论）
   - 评论按正序展示（最早在前）
   - 采用扁平化卡片式设计，回复内嵌在主评论下方
   - **评论直接显示在动态卡片下方（QQ空间风格）**
   - **点击评论图标弹出输入框，仅用于发表评论，不打开全屏面板**

2. **评论点击交互（QQ空间风格）**
   - **点击自己的评论 → 弹出操作菜单：删除、取消**
   - **点击别人的评论 → 弹出输入框，回复该评论**
   - **点击其他区域 → 关闭输入框/菜单**

3. **通知功能**
   - 在动态页面右上角添加通知图标
   - 通知图标显示红点徽章（有新通知时）
   - 点击通知图标进入通知页面
   - 通知页面展示未读评论通知列表
   - 支持点击单条通知标记为已读
   - 支持从通知跳转到对应的动态
   - 通知类型仅包含评论和回复

3. **UI设计**
   - 全新设计评论组件，参考QQ空间扁平化卡片风格
   - 优化评论展示层次和交互体验

### 步骤八：在动态页面添加通知图标

**文件**: `frontend/client/src/components/ui/DesktopMenu.vue`

**现状分析**:
当前页面右上角已有 DesktopMenu 组件（fixed top-8 right-6），用于服务器切换。

**页面结构**:
```
┌─────────────────────────────────────────────────────┐
│ 标题区域 (max-w-6xl mx-auto px-6)                    │
│ ├─ 标题: "时光动态"                                   │
│ └─ 副标题: "记录我们的点点滴滴"                        │
├─────────────────────────────────────────────────────┤
│ DesktopMenu (fixed top-8 right-6) ← 右上角固定       │
├─────────────────────────────────────────────────────┤
│ 主内容区域                                           │
└─────────────────────────────────────────────────────┘
```

**修改方案**:
将 DesktopMenu 改造为右上角功能区，包含通知图标 + 菜单按钮：

```
┌──────────────────────────┐
│ [通知图标] [菜单按钮]     │  ← 右上角固定区域
└──────────────────────────┘
```

**改动说明**:
- 修改 `DesktopMenu.vue`，添加通知图标按钮（在菜单按钮左侧）
- 通知图标显示红点徽章（有未读通知时）
- 点击通知图标跳转到 `/notifications` 页面
- 保持原有菜单按钮功能不变

**UI设计**:
- 通知图标和菜单按钮水平排列，间距 8px
- 两个按钮使用相同的玻璃态样式
- 红点徽章在通知图标右上角（top-1 right-1）
- 整体容器固定在 top-8 right-6

**优点**:
1. 不破坏现有布局结构
2. 右上角功能集中，用户体验一致
3. 复用 DesktopMenu 的玻璃态样式

### 步骤九：后端API支持

**状态**: 需要新增

**文件**: 
- `backend/internal/model/notification.go` - 通知模型
- `backend/internal/repository/notification.go` - 通知数据访问层
- `backend/internal/service/notification.go` - 通知业务逻辑
- `backend/internal/handler/notification.go` - 通知HTTP处理器
- `backend/internal/router/router.go` - 路由注册

**改动说明**:
- 创建通知数据库表结构
- 实现通知相关API端点
- 在评论创建时自动生成通知
- 实现通知已读状态更新

**数据库表结构**:
```sql
CREATE TABLE notifications (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,          -- 接收通知的用户ID
  sender_id INT NOT NULL,        -- 发送通知的用户ID
  moment_id INT NOT NULL,        -- 关联的动态ID
  comment_id INT NOT NULL,       -- 关联的评论ID
  type VARCHAR(20) NOT NULL,     -- 类型: 'comment' | 'reply'
  content TEXT,                  -- 通知内容（评论内容摘要）
  is_read BOOLEAN DEFAULT FALSE, -- 是否已读
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (sender_id) REFERENCES users(id),
  FOREIGN KEY (moment_id) REFERENCES moments(id),
  FOREIGN KEY (comment_id) REFERENCES comments(id)
);
```

**API端点**:
- `GET /api/notifications/unread` - 获取未读通知
- `POST /api/notifications/:id/read` - 标记单条已读
- `GET /api/notifications/count` - 获取未读数量
- `POST /api/notifications/read-all` - 标记全部已读

**业务逻辑**:
- 当用户A评论用户B的动态时，给用户B发送评论通知
- 当用户A回复用户B的评论时，给用户B发送回复通知
- 通知生成在评论创建的事务中完成

### 步骤十二：评论点击交互（QQ空间风格）

**文件**: `frontend/client/src/components/comment/CommentItem.vue`

**改动说明**:
- 评论内容区域可点击
- 根据是否是自己的评论，执行不同操作：

**交互逻辑**:

| 点击对象 | 触发行为 |
|----------|----------|
| 自己的评论 | 弹出 ActionSheet 菜单（删除、取消） |
| 别人的评论 | 弹出输入框，回复该评论 |
| 其他区域 | 关闭输入框/菜单 |

**实现方式**:
1. 点击评论内容 → 判断 `comment.userId === currentUserId`
2. 如果是自己的评论 → 显示 ActionSheet（删除、取消）
3. 如果是别人的评论 → emit('reply', comment) → 父组件弹出输入框
4. 点击输入框/菜单外部区域 → 关闭

**组件改动**:
1. CommentItem: 点击评论内容时判断身份并触发不同事件
2. CommentList: 接收 reply 事件，管理输入框显示
3. MomentsView: 管理全局评论输入框状态
